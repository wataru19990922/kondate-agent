# AGENT_SCENARIOS.md

献立提案エージェントの **会話フロー例**。

各シナリオは:
- ユーザの典型的な発話パターン
- エージェントの内部判断 (どのサブエージェントを呼んだか、何を判断材料にしたか)
- 最終応答

を示す。本ドキュメントは:
1. 設計の妥当性検証
2. プロンプトチューニングの教材
3. デモ用シナリオ台本

を兼ねる。

---

## 前提となるユーザ状態 (シナリオ共通)

```yaml
user_profile:
  household_size: 2
  allergies: []
  dietary_prefs: []
  protein_target_per_day: 60  # g (大人 1 人分)

inventory:  # 在庫 (今日時点)
  - 鶏もも肉: 300g
  - 玉ねぎ: 2 個
  - 卵: 6 個
  - にんじん: 1 本
  - ピーマン: 3 個 (期限 2 日後)
  - 長ナス: 2 本 (期限 1 日後)
  - 木綿豆腐: 1 丁 (期限 3 日後)
  - 醤油: 500ml
  - 砂糖: 1kg

meal_history_last_7days:
  - 月: 鶏むね肉のチキン南蛮 (P:32g)
  - 火: 鮭のホイル焼き (P:26g)
  - 水: 豚肉と野菜の生姜焼き (P:28g)
  - 木: 牛丼 (P:35g)
  - 金: 麻婆豆腐 (P:22g)
  - 土: サラダうどん (P:14g)
  - 日: (まだ未決定)
```

## シナリオ 1: 完全に開かれた質問

**ユーザ**: 「今夜何食べる?」

### 内部処理

```
[1] Coordinator が状況を把握する必要を判断
    → 「ユーザは未決定。在庫・履歴・栄養を総合判断して提案を作る」

[2] get_inventory_summary()
    → {meat: ['鶏もも肉 300g'], veg: ['玉ねぎ', 'ピーマン', '長ナス', 'にんじん'], 
       protein: ['卵', '木綿豆腐'], expiring: ['長ナス']}

[3] check_meal_variety(days=7)
    → HistoryAnalyst が分析
    → {
        recent_pattern: '肉中心 (5/6 日)',
        protein_avg: 26.2 g/day,  # 目標 60g に対して未達
        underused_categories: ['野菜単品'],
        underused_inventory: ['長ナス (期限明日)', '木綿豆腐']
      }

[4] Coordinator の判断:
    - タンパク質目標未達 → 高タンパクな主菜
    - 期限近 → 長ナス を使うものを優先
    - 肉続き → 軽めの調理 or 豆腐系もアリ

[5] find_recipes(query="鶏もも肉 長ナス または 豆腐 主菜", filters={time: "30分以内"})
    → 候補 4 件

[6] analyze_nutrition(top_3, household_size=2)
    → {
        '鶏もも肉と長ナスの炒め物': {P: 28g, target_contribution: '+47%'},
        '木綿豆腐と鶏ひき肉のあんかけ': {P: 24g, target_contribution: '+40%'},
        ...
      }

[7] Coordinator が応答を組み立てる
```

### エージェントの最終応答

> 今週お肉が続いてて (6 日中 5 日)、タンパク質も目標の 60g に対して平均 26g と少なめでした。
> 在庫の **長ナス (期限明日)** と **鶏もも肉** で、タンパク質をしっかり摂れる候補を 3 つ。
>
> 1. **鶏もも肉と長ナスの中華風炒め** (15 分・在庫 OK・タンパク質 28g/人)
> 2. **鶏もも肉のソテー 長ナス添え** (20 分・在庫 OK・タンパク質 30g/人)
> 3. **長ナスと豚バラ肉の味噌炒め** (15 分・豚バラ不足 → 鶏もも代用可)

## シナリオ 2: 曖昧な気分 (「あっさり系」)

**ユーザ**: 「あっさり系で」

### 内部処理

```
[1] Coordinator: 「あっさり = 油・肉が控えめ・野菜中心」と解釈

[2] check_meal_variety(days=7)
    → 「肉が連続」「油の多い料理が続いてる」→ あっさり方向は妥当

[3] find_recipes(query="あっさり 野菜中心 さっぱり",
                 filters={exclude_categories: ['揚げ物', 'こってり系']})
    → 候補 5 件

[4] match_inventory()
    → 候補 5 件の在庫充足率を計算
    → ['長ナスとピーマンの焼き浸し' (在庫 100%), 
        '木綿豆腐のサラダ' (在庫 90%),
        '玉ねぎとにんじんのマリネ' (在庫 100%)]
        ...

[5] analyze_nutrition(top_3, household_size=2)
    → ただし「あっさり」だとタンパク質低めになりがちなので、
      Coordinator が判断: 「タンパク質を補える組み合わせ」を提案するか主菜選びを考える

[6] Coordinator が応答に「タンパク質補強の選択肢」を含めて返す
```

### エージェントの最終応答

> あっさり系ですね。在庫の長ナスとピーマン (期限近め) を活かせる候補です。
>
> 1. **長ナスとピーマンの焼き浸し** (10 分・在庫 100%・タンパク質 6g/人)
> 2. **木綿豆腐とにんじんのサラダ** (15 分・在庫 95%・タンパク質 14g/人)
> 3. **玉ねぎと卵のさっぱり炒め** (10 分・在庫 100%・タンパク質 12g/人)
>
> 今週タンパク質が少なめなので、もし副菜なら 1 + 2 の組み合わせがバランスいいです。

## シナリオ 3: 具体的な制約付き

**ユーザ**: 「子供が魚嫌いなので魚以外で。30 分以内に食べたい。」

### 内部処理

```
[1] Coordinator: 制約を抽出
    → constraints = {exclude_ingredients: ['魚', '鮭', 'サバ', '...'],
                     max_cooking_time: 30 minutes}

[2] check_meal_variety(days=7)
    → 「肉中心、魚 1 日 (月の鮭)」→ HistoryAnalyst としては魚を勧めたいが、
      Coordinator が制約優先と判断
    → 「肉中心のなかでもジャンルを変える」方向に補正

[3] find_recipes(query="鶏 または 豚 または 卵", 
                 filters={exclude: ['魚'], time_max: 30, in_stock_only: false})
    → 候補 6 件

[4] Coordinator: ジャンル多様性チェック
    → 直近: 鶏南蛮(揚げ)、生姜焼き(炒め)、牛丼(丼)、麻婆豆腐(中華)
    → 「和食 (煮物 or 焼き)」が抜けている
    → 候補から「和の焼き物」を優先

[5] analyze_nutrition(top_3)
    → 全候補タンパク質 25g+ → OK

[6] 最終応答
```

### エージェントの最終応答

> 魚以外・30 分以内ですね。今週はジャンルが揚げ・炒め・中華に寄ってたので、和の焼き物系を中心に。
>
> 1. **鶏もも肉の照り焼き** (20 分・在庫 OK・タンパク質 30g/人)
> 2. **木綿豆腐と卵のチャンプルー** (15 分・在庫 OK・タンパク質 22g/人)
> 3. **鶏ひき肉のそぼろ丼** (15 分・鶏ひき肉不足・タンパク質 28g/人)
>
> お子さんに食べてもらいやすい味付け重視で選びました。

## シナリオ 4: 既存の提案からの絞り込み

**前ターン**: シナリオ 1 の結果 (候補 3 件) が表示済み

**ユーザ**: 「もっと早くできるのある?」

### 内部処理

```
[1] Coordinator: 「前回の候補に対する絞り込み」と解釈
    → 会話履歴から前回の 3 候補を取得
    → 各々の調理時間: [15分, 20分, 15分]
    → 「もっと早く」= 10 分前後を狙う

[2] find_recipes(query="鶏もも肉 長ナス 簡単", 
                 filters={time_max: 10, exclude_recipe_ids: [前回 3 件]})
    → 候補 3 件

[3] analyze_nutrition(top_3)
    → タンパク質はやや下がる (20-25g)
    → Coordinator が応答にトレードオフ言及
```

### エージェントの最終応答

> 10 分以内だとこちら。さっきよりタンパク質はやや下がるけど時短重視で。
>
> 1. **長ナスとピーマンの簡単炒め** (8 分・在庫 OK・タンパク質 12g/人)
> 2. **鶏もも肉のレンジ蒸し** (10 分・在庫 OK・タンパク質 26g/人)
> 3. **卵と野菜のスクランブル** (5 分・在庫 OK・タンパク質 14g/人)
>
> タンパク質しっかりなら 2 番がおすすめ。

## シナリオ 5: 食材指定

**ユーザ**: 「長ナス使いきりたい」

### 内部処理

```
[1] Coordinator: 「特定食材を使う」と解釈
    → 在庫の長ナスは 2 本
    → 期限が明日 → 優先度高い

[2] find_recipes(query="長ナス", filters={contains: ['長ナス']})
    → 候補 5 件
    → 在庫充足率でソート

[3] check_meal_variety(days=7) (軽い参照)
    → 「ジャンルの多様性は意識するが、食材指定が優先」

[4] analyze_nutrition(top_3) (副次的)
```

### エージェントの最終応答

> 長ナス 2 本ですね。明日が期限なので使い切り方向で。
>
> 1. **長ナスの煮浸し** (15 分・在庫 OK・長ナス 2 本フル消費)
> 2. **長ナスと鶏もも肉の炒め物** (15 分・在庫 OK・タンパク質 28g/人)
> 3. **麻婆ナス** (20 分・豚ひき肉不足・長ナス 2 本フル消費)
>
> タンパク質も考えるなら 2 番、長ナス主役なら 1 番が映えます。

## シナリオ 6: 週間プランニング (将来機能)

**ユーザ**: 「今週の献立まとめて決めたい」

### 内部処理

```
[1] Coordinator: 「複数日提案モード」に切替
    → 「明日から 3 日分」程度に縮約 (週 7 日は重いので段階的に)

[2] check_meal_variety(days=14) (より長期参照)
    → 過去の傾向 + 直近の偏り

[3] 3 日分のプランを立てる:
    Day 1: タンパク質多め + 期限近食材消費
    Day 2: 野菜多め (Day 1 と対比)
    Day 3: 簡単料理 (週後半は時短)

[4] 各 Day で find_recipes(filters=各 Day の条件)
[5] analyze_nutrition() を 3 日合計で評価
[6] Coordinator: 「在庫の使用量が日別に整合してるか」最後にチェック
   (例: にんじんを 3 日とも使ってしまわないか)
```

### エージェントの最終応答

> 3 日分のプランを提案します。在庫使い切り + 栄養バランスを意識:
>
> **Day 1 (今日)**: 鶏もも肉と長ナスの炒め (期限近食材消化、タンパク質しっかり)
> **Day 2 (明日)**: 木綿豆腐と野菜の煮物 (軽め、Day 1 の反動で野菜中心)
> **Day 3 (明後日)**: 卵と玉ねぎのオムレツ + 簡単サラダ (時短・低コスト)
>
> 3 日合計タンパク質 84g/人 (目標 180g に対して 47%)。残りは買い足し or 別途補強で。

## シナリオ 7: 在庫不足の場合

**ユーザ**: 「在庫だけで作れるやつ」(在庫がほぼ空の状態)

### 内部処理

```
[1] get_inventory_summary()
    → 在庫が 2 品しかない (卵 6 個・醤油のみ)

[2] find_recipes(filters={in_stock_only: true})
    → 候補 0 件 ! 

[3] Coordinator: 「完全在庫マッチが無理」と判断
    → 候補緩和: 「在庫の 80% 以上カバー」に下げる
    → それでも 1 件のみ

[4] 提案戦略変更: 「在庫メイン + 買い足し 1〜2 品」を明示的に提案
    → find_recipes(query="卵 卵料理", filters={max_missing: 2})
    → 候補 3 件

[5] 応答に「買い足しリスト」を含める
```

### エージェントの最終応答

> 在庫だけだと候補が厳しいですね…。卵 6 個があるので、卵を主役にした候補を 3 つ。少しだけ買い足しが必要なものも含めました。
>
> 1. **卵焼き** (5 分・在庫 100%・タンパク質 12g/人) ← 完全に在庫だけで OK
> 2. **オムレツ** (10 分・牛乳少量と粉末コンソメ要・タンパク質 14g/人)
> 3. **親子丼** (20 分・玉ねぎと鶏肉が必要・タンパク質 26g/人)
>
> もう少し本格的に作るなら、近所のスーパーで **鶏もも肉 1 パック + 玉ねぎ** を買えば親子丼が映えます。

## 共通パターン分析

各シナリオから抽出した、エージェントの **必須の振る舞い**:

| 振る舞い | シナリオ | 説明 |
|---|---|---|
| **履歴を意識した提案** | 1, 2, 3 | HistoryAnalyst を呼んで偏りに言及 |
| **栄養目標との差分言及** | 1, 2, 3, 5 | NutritionAnalyst が「+47%」など具体数字 |
| **期限近食材の優先消費** | 1, 5 | 在庫の expiring_soon を必ずチェック |
| **制約の絶対遵守** | 3 | 制約 > 履歴インサイト |
| **会話継続の文脈** | 4 | 直前の提案を覚えて絞り込む |
| **代替案の提示** | 3, 7 | 厳しい場合は緩和案を提案 |
| **トレードオフの明示** | 4, 7 | "タンパク質は下がるけど時短" |
| **数量の具体性** | 全部 | "300g", "30g/人", "15 分" など曖昧表現を避ける |

## システムプロンプトに盛り込むべき要素 (草稿)

```
あなたは家庭の献立提案エージェントです。

## 振る舞いの原則
1. 必ず HistoryAnalyst を呼んで直近の偏りを確認すること。提案文に偏りへの言及を 1 文入れる
2. 在庫の expiring_soon は優先的に消化する候補を含める
3. 栄養目標との差分は数字で示す ("+47%" 等)
4. ユーザの制約 (アレルギー、嫌い、時間) は絶対遵守
5. 候補は最大 3 件。理由付きで提案
6. 完全在庫マッチが無理なら、必要買い足しを明示

## やってはいけないこと
- 嘘の栄養数値を出す (推定であることを明示)
- ユーザが嫌いと言ったものを「でも栄養的に」と再提案
- 在庫にない食材を「在庫あり」として扱う
- 候補を 5 件以上出して選ぶ気を失わせる

## 出力スタイル
- 短い前置き (1〜2 文) で判断理由を示す
- 候補は番号付き 1〜3 件
- 各候補に「調理時間・在庫充足・タンパク質量」を必ず併記
```

## デモシナリオとしての推奨組み合わせ

**3 分デモ用** (シナリオ 1 → 4 → 5 の流れ):

1. 「今夜何食べる?」 (シナリオ 1)
   → 履歴分析・栄養補正・期限消化の判断が全部見える。**メインの自慢ポイント**
2. 「もっと早くできるのある?」 (シナリオ 4)
   → 会話継続性・絞り込み能力をアピール
3. 「長ナス使いきりたい」 (シナリオ 5)
   → ユーザ主導の制約に応答する柔軟性

合計 90 秒程度のやり取り。3 つのサブエージェントが全部登場する。
